# Expected Q&A — pp_cotrain

Rank-ordered from most-likely-asked to edge cases. Each entry has a **short answer** (1–2 sentences for verbal response) and **follow-up detail** (if they push).

---

## Q1. Why IQL and not MAPPO / QMIX / VDN?

**Short**: Because the point of a baseline is to isolate variables. IQL has no opponent-modeling, no value-decomposition, no CTDE assumption — so any lift from our proposed method is clearly attributable to the opponent-reward modeling, not to "better credit assignment" or "centralized training".

**Detail**: MAPPO is a policy-gradient method and would change the value-function story entirely. QMIX and VDN assume a cooperative centralized Q — they make an IGM-style monotonicity assumption that doesn't even hold across adversarial teams (the team global Q is not a monotone function of opponent local Qs). So QMIX/VDN are not even definable across two adversarial teams; they can only be used within a team. If you want an intra-team credit-assignment baseline, we can add QMIX *within* the predator team as a second baseline.

---

## Q2. Why is this "co-training" and not just "two IQL agents"?

**Short**: Co-training here means both teams' policies are learning *simultaneously* against each other, generating non-stationary target distributions for one another's Bellman updates. That's exactly the setting the proposed paper targets.

**Detail**: If we held one team fixed (e.g. scripted prey) and trained the other, we'd be doing a standard single-agent RL problem with a fixed adversary — easier, but not the regime we care about. The co-training dynamics visible in the return curves (pred up → prey adapts → pred drops) is evidence that neither side is stationary.

---

## Q3. Why share parameters within the predator team?

**Short**: The 3 predators are interchangeable — same role, same obs structure, same reward function. Sharing params cuts sample complexity by 3× and is standard in cooperative MARL (VDN, MAPPO, QMIX all share within-team).

**Detail**: The agent-ID one-hot (added by `CTRolloutManager`) gives each predator a distinguishing identity in its observation, so the shared network can still produce different actions for different predators when state asymmetries demand it. For the MPE simple_tag task all 3 predators are actually symmetric, so the ID one-hot is mostly redundant — it's there for generality.

---

## Q4. What's the capture rate? How do we know predators actually learned?

**Short**: Last greedy eval was **+33.125 predator return per 30-step episode**; +10 per predator per capture × 3 predators = +30 per capture event, so ~1.1 captures per 30-step rollout. Started at +14.375, so we more than doubled.

**Detail**: Train-time rollout with seed-7 greedy: **5 captures in 60 steps**, one every ~12 steps. Exploration is ε=0.05 at the end, so some stochasticity remains; greedy eval is the fair number.

---

## Q5. Why does the predator return *drop* after peaking at step ~2000?

**Short**: That's prey catching up. It's the expected signature of adversarial co-training: when one side temporarily has a better policy, the other side's gradients get a stronger learning signal and it closes the gap. The equilibrium is ±2, not the early peak.

**Detail**: In fact, the fact that pred peaks then settles is reassuring — if pred had kept climbing monotonically to +5, that would mean prey is failing to learn, which would be a bug. The non-monotonic return curve is how you see co-adaptation.

---

## Q6. Why γ=0.9? That's short.

**Short**: Default MPE episode length is 25 steps, so γ^25 ≈ 0.07 already heavily discounts anything beyond the episode boundary. γ=0.99 would imply looking ~100 steps ahead and doesn't match the task horizon.

**Detail**: This is the JaxMARL default for MPE and matches the PyMARL baselines. Running with γ=0.99 is easy if we want to try; I don't expect it to change qualitative conclusions.

---

## Q7. Why only 1 seed? What's the variance?

**Short**: The 2 M run takes 150 s on CPU — I can launch multiple seeds right now if you want. The `iql_teams.py` script already supports `NUM_SEEDS=N` via `jax.vmap`, so it's a flag change, not a code change.

**Detail**: Launching with `NUM_SEEDS=5` would take ~5× wall-clock or less due to JAX's parallel tracing on vmap; still under ~15 minutes. Reasonable budget before any paper submission.

---

## Q8. Why not WandB?

**Short**: Deliberately offline for this run — everything is saved as `.npz` and plotted locally so the artifacts are deterministic and self-contained in the repo. WandB is a one-liner: `WANDB_MODE=online` in the config.

---

## Q9. Why Q-values end up at +5 / −5?

**Short**: With γ=0.9, an expected reward of +30 per capture occurring `n` steps in the future discounts to `30 × 0.9^n`. For Q ≈ +5, `n ≈ log(5/30) / log(0.9) ≈ 17` steps to expected capture. Matches the observed ~1 capture per ~25 steps.

**Detail**: This is a consistency check, not a proof — real Q includes variance across states — but the magnitude is right. If Q had converged to 0 that would mean the predator doesn't expect captures, which would contradict the +33 test return.

---

## Q10. Why isn't there a centralized critic / global state?

**Short**: This is *Independent* Q-Learning — the "I" in IQL. Each agent sees only its own observation. That's the baseline; centralized critics are in QMIX / MADDPG / MAPPO and would be separate comparisons.

**Detail**: The stock IQL uses a global state too (via `obs["__all__"]`), but the network it feeds doesn't use it; it only feeds the per-agent observation. We keep the same.

---

## Q11. What happens at `LEARNING_STARTS=10000`?

**Short**: For the first 10 000 env-steps the policy is pure ε=1.0 random exploration to fill the replay buffer. No gradient updates happen until the buffer has enough diversity for stable Bellman regression.

---

## Q12. Why are obs sizes different (16 vs 14)?

**Short**: Predator obs includes 2 extra dims for the prey's velocity (predator can observe what the prey is doing); prey obs omits predator velocity because in the MPE default, prey can only see predator positions, not velocities. The `CTRolloutManager` wrapper pads both to the max (16) and adds a 4-dim agent-ID one-hot.

**Detail**: This is a PettingZoo convention. It means the prey has slightly *less* information than the predators — an additional asymmetry baked into the task.

---

## Q13. Is the prey allowed to leave the arena?

**Short**: Yes, but it's penalized smoothly via `map_bounds_reward`. The prey learned to stay in; note the test return improved from −247 to −49, which is almost entirely boundary-penalty reduction, not capture avoidance.

**Detail**: Predators cannot leave the arena — the MPE physics treat the arena as a bounded region only for good agents in `simple_tag`.

---

## Q14. What makes this better than behavior cloning the opponent?

**Short**: Behavior cloning fits a classifier to `opponent_action | state`. If the opponent changes policy, the BC target moves with it — you're always one step behind. Our proposed method learns the opponent's *reward*, which is invariant under policy changes as long as the opponent's goal is stable.

**Detail**: This is the core research bet: a reward is a more compact and more stable latent variable than a policy. BIRL gives us a Bayesian posterior over reward hypotheses, not a point estimate, so we also get uncertainty about the opponent's objective.

---

## Q15. What if the PI asks "can we train in SMAC / Starcraft instead"?

**Short**: SMAC has continuous state, discrete actions, and is asymmetric, so it fits the desiderata. The scale-up costs are manageable because JaxMARL runs SMAX (the JAX re-implementation of SMAC) on GPU. But we chose simple_tag first because it lets us iterate on the algorithm in seconds, not hours.

**Detail**: Once the model-based method is working on simple_tag, moving to SMAX is mostly a config swap — `iql_rnn.py` already supports `ENV_NAME: SMAX_*` in stock JaxMARL. Our two-team split logic would need to handle SMAX's team structure, which is different (both sides can have multiple units of heterogeneous types).

---

## Q16. Why flashbax? Why not a simple uniform replay buffer?

**Short**: flashbax is the JAX-native buffer library, JIT-compatible, and the entire JaxMARL codebase uses it. Writing our own would work but be slower and re-implement what flashbax already does.

---

## Q17. What's the next concrete deliverable after this baseline?

**Short**: Two tracks in parallel: (1) implement the BIRL reward estimator — likely GPNN or a small reward MLP trained on opponent trajectories; (2) wire an MCTS planner that uses that estimator to roll out hypothetical games.

**Detail**: MCTS for 3 predators × 5 actions = 125 joint actions per tree node — borderline but tractable with progressive widening. The dec-MCTS literature (IJCAI 2020) gives us a factored alternative if the joint action space blows up.
